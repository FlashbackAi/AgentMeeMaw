# SCHEMA.md — Flashback AI: Legacy Mode

This is the column-by-column reference for the canonical graph. Read
`ARCHITECTURE.md` for the system context and `CLAUDE.md` for the rules
of engagement.

---

## 1. Overview

- **Engine:** Postgres + pgvector.
- **Model:** strongly-typed node tables + one generic `edges` table.
- **Lifecycle:** `status` columns rather than hard deletes. Always
  query through the `active_*` views.
- **IDs:** UUIDv4 (`gen_random_uuid()` from pgcrypto).
- **Time:** `TIMESTAMPTZ`, defaulted to `now()`. `updated_at` is
  auto-maintained by the `trg_set_updated_at` trigger on every
  audited table.
- **Embeddings:** `vector(1024)` from pgvector, paired with
  `embedding_model` + `embedding_model_version` columns. CHECK
  constraints enforce that the three move together. HNSW indexes are
  partial — built only on `status = 'active'` rows.

The agent is the **only** writer to these tables. Node reads them for
the UI but never writes. See `CLAUDE.md` §3.

## 2. Node tables

### 2.1 `persons`

The subject of a legacy. One row per legacy.

| Column | Type | Notes |
|---|---|---|
| `id` | UUID PK | `gen_random_uuid()` |
| `name` | TEXT NOT NULL | |
| `relationship` | TEXT | Contributor's relationship label (e.g. "father") |
| `profile_summary` | TEXT | Generated by Profile Summary Generator |
| `phase` | TEXT NOT NULL DEFAULT `'starter'` | CHECK in (`'starter'`, `'steady'`). Sticky. |
| `coverage_state` | JSONB NOT NULL DEFAULT `{sensory:0,voice:0,place:0,relation:0,era:0}` | 5 anchor dims |
| `phase_locked_at` | TIMESTAMPTZ | Set when Handover Check fires |
| `moments_at_last_thread_run` | INT NOT NULL DEFAULT 0 | Drives Thread Detector cadence |
| `image_url` | TEXT | Written by Node consumer of the artifact queue |
| `thumbnail_url` | TEXT | Written by Node consumer |
| `generation_prompt` | TEXT | Written by **agent**, consumed by Node |
| `created_at`, `updated_at` | TIMESTAMPTZ | Auto-maintained |

`active_persons` view exists for API symmetry but is just `SELECT *`.

**No `date_of_birth` / `date_of_death`.** Cold UX for grief; lifespan
is derived from moment time anchors.

### 2.7 `profile_facts`

Open-ended `(question, answer)` pairs surfaced on the legacy profile
page. The seven seed slugs are display defaults, not a hard registry.

| Column | Type | Notes |
|---|---|---|
| `id` | UUID PK | |
| `person_id` | UUID NOT NULL FK→persons | ON DELETE CASCADE |
| `fact_key` | TEXT NOT NULL | snake_case slug. Free-form; LLM- or Node-supplied. Seeds: `profession`, `birthplace`, `residence`, `faith`, `family_role`, `era`, `personality_essence` |
| `question_text` | TEXT NOT NULL | How the fact reads on the profile (with name interpolated) |
| `answer_text` | TEXT NOT NULL | 1–15 words typically; cap is 300 chars |
| `source` | TEXT NOT NULL | `starter_extraction` (profile_summary worker) or `user_edit` (`POST /profile_facts/upsert`) |
| `status` | TEXT NOT NULL DEFAULT `'active'` | `'active'` or `'superseded'` |
| `superseded_by` | UUID NULL FK→profile_facts(id) | |
| `answer_embedding` | vector(1024) | Filled by embedding worker |
| `embedding_model`, `embedding_model_version` | TEXT | Lockstep with vector |
| `created_at`, `updated_at` | TIMESTAMPTZ | |

**Constraints:**
- `chk_profile_facts_embedding_complete` — vector + model identity move together.

**Indexes:**
- Partial UNIQUE `(person_id, fact_key) WHERE status='active'` — at
  most one active row per (person, key).
- `(person_id, status)` — cap-counting + per-person listing.

**Cap:** 25 active rows per person, enforced in app code
(`flashback.profile_facts.repository.MAX_ACTIVE_FACTS_PER_PERSON`).
Updates to existing keys are always allowed; new keys are rejected at
the cap.

**View:** `active_profile_facts` filters to `status='active'`.

## 3. Review / workflow tables

### 3.1 `identity_merge_suggestions`

Pending user-review items for entity identity corrections. The
Extraction Worker or identity-merge scanner may create suggestions
automatically, but neither path merges entities. Approval/rejection is an explicit API/UI action
(`POST /identity_merges/suggestions/{id}/approve|reject`).

Typical case:

```text
source_entity: "old label for Person B"
target_entity: "Person B"
proposed_alias: "old label for Person B"
status: pending
```

| Column | Type | Notes |
|---|---|---|
| `id` | UUID PK | |
| `person_id` | UUID NOT NULL FK→persons | Legacy scope |
| `source_entity_id` | UUID NOT NULL FK→entities | Losing entity if approved |
| `target_entity_id` | UUID NOT NULL FK→entities | Surviving entity if approved |
| `proposed_alias` | TEXT | Label to fold into target aliases |
| `reason` | TEXT NOT NULL | Human-readable review reason |
| `source` | TEXT NOT NULL DEFAULT `'extraction'` | CHECK in (`'extraction'`, `'scanner'`, `'user_edit'`, `'admin'`) |
| `status` | TEXT NOT NULL DEFAULT `'pending'` | CHECK in (`'pending'`, `'approved'`, `'rejected'`) |
| `approved_at`, `rejected_at` | TIMESTAMPTZ | Mutually tied to status by CHECK constraints |
| `created_at`, `updated_at` | TIMESTAMPTZ | Auto-maintained |

**Constraints and indexes:**
- `source_entity_id <> target_entity_id`
- partial UNIQUE `(person_id, source_entity_id, target_entity_id) WHERE status='pending'`
- `(person_id, status, created_at DESC)`

**View:** `pending_identity_merge_suggestions` filters to
`status='pending'`.

Approval effects are intentionally outside this table: the approval
handler marks the source entity `merged`, sets `merged_into`, repoints
entity edges, updates target aliases/description, clears target
embedding fields, and queues a fresh entity embedding job.

## 4. The `edges` table

| Column | Type | Notes |
|---|---|---|
| `id` | UUID PK | |
| `from_kind` | TEXT NOT NULL | CHECK in 6 kinds |
| `from_id` | UUID NOT NULL | **Not** a FK (heterogeneous) |
| `to_kind` | TEXT NOT NULL | CHECK in 6 kinds |
| `to_id` | UUID NOT NULL | **Not** a FK |
| `edge_type` | TEXT NOT NULL | CHECK in 8 types |
| `attributes` | JSONB NOT NULL DEFAULT `'{}'` | Edge-specific data (e.g., role) |
| `status` | TEXT NOT NULL DEFAULT `'active'` | CHECK in (`'active'`, `'archived'`) |
| `created_at`, `updated_at` | TIMESTAMPTZ | |

**UNIQUE** on `(from_kind, from_id, to_kind, to_id, edge_type)` —
prevents exact duplicate edges.

**Indexes:**
- `(from_kind, from_id, edge_type, status)` — outbound
- `(to_kind, to_id, edge_type, status)` — inbound
- `(edge_type, status)` — type-wide scans

### 4.1 Edge type matrix

App-level `validate_edge()` (in `src/db/edges.py`) enforces these
combinations. The DB only checks `edge_type` membership.

| `edge_type` | Allowed `(from_kind → to_kind)` | Notes |
|---|---|---|
| `involves` | moment → entity | Role goes in `attributes` |
| `happened_at` | moment → entity | Entity must be `kind='place'` (caller-enforced) |
| `exemplifies` | moment → trait | Evidence link |
| `evidences` | moment → thread, entity → thread, thread → trait, entity → trait | thread → trait written by Trait Synthesizer; entity → trait is forward-compat (no v1 writer) |
| `related_to` | entity → entity | Generic relationship |
| `motivated_by` | question → moment, question → entity, question → thread | What seeded this question |
| `targets` | question → entity | What the question is asking about |
| `answered_by` | question → moment | Filled by Extraction Worker when a seeded segment becomes a moment |

**Caller responsibilities (not enforced by `validate_edge()`):**
- Both endpoints must belong to the same `person_id`. (Exception:
  `from_kind='question' AND from.person_id IS NULL` is allowed when
  the question is a `starter_anchor` template — the answered moment's
  `person_id` is the scope.)
- Both endpoints must be `status='active'`.
- For `happened_at`: the target entity must be `kind='place'`.

## 7. Embedding completeness invariant

Every embedded table has a CHECK constraint of the form:

```sql
(vector_col IS NULL
    AND embedding_model IS NULL
    AND embedding_model_version IS NULL)
OR
(vector_col IS NOT NULL
    AND embedding_model IS NOT NULL
    AND embedding_model_version IS NOT NULL)
```

**Why:** writers INSERT with the vector NULL and the model fields
also NULL. The embedding worker UPDATEs all three together once it
has the vector from Voyage. We never see "vector set but no model"
or vice versa, which would silently break similarity searches across
model upgrades (invariant #3).

## 8. Common query patterns

These are the read patterns the agent and Retrieval Service rely on.
Always include `status='active'` (or use the views) and
`person_id = $1` (legacy scope, invariant #2).

### Vector search: moments

```sql
SELECT id, title, narrative,
       narrative_embedding <=> $2 AS distance
FROM   active_moments
WHERE  person_id = $1
  AND  embedding_model = $3
  AND  embedding_model_version = $4
ORDER  BY narrative_embedding <=> $2
LIMIT  10;
```

The `embedding_model` filter prevents cross-model contamination
(invariant #3).

### Vector search: entities

```sql
SELECT id, kind, name, description
FROM   active_entities
WHERE  person_id = $1
  AND  embedding_model = $3 AND embedding_model_version = $4
ORDER  BY description_embedding <=> $2
LIMIT  10;
```

### All entities of a kind for a person

```sql
SELECT * FROM active_entities
WHERE person_id = $1 AND kind = $2;
```

### Moments mentioning a given entity

```sql
SELECT m.*
FROM   active_moments m
JOIN   active_edges e
  ON   e.from_kind = 'moment'
  AND  e.from_id   = m.id
  AND  e.edge_type = 'involves'
WHERE  e.to_kind = 'entity'
  AND  e.to_id   = $1
  AND  m.person_id = $2;
```

### Threads an entity contributes to

```sql
SELECT t.*
FROM   active_threads t
JOIN   active_edges e
  ON   e.to_kind = 'thread' AND e.to_id = t.id
WHERE  e.from_kind = 'entity'
  AND  e.from_id   = $1
  AND  e.edge_type = 'evidences'
  AND  t.person_id = $2;
```

### Open dropped-reference questions for a session

The agent stitches `session_id` via the seeding question's
`motivated_by` edges; the simplest scoped query is per-person.

```sql
SELECT id, text, attributes->>'dropped_phrase' AS phrase
FROM   active_questions
WHERE  person_id = $1
  AND  source    = 'dropped_reference'
  AND  status    = 'active';
```

### Phase Gate: pick a starter for a dimension

```sql
SELECT id, text
FROM   active_questions
WHERE  source = 'starter_anchor'
  AND  attributes->>'dimension' = $1   -- 'sensory' | 'voice' | ...
  AND  NOT EXISTS (
        SELECT 1 FROM active_edges e
         WHERE e.from_kind = 'question' AND e.from_id = active_questions.id
           AND e.edge_type = 'answered_by'
           AND e.to_kind  = 'moment'
           AND EXISTS (
                SELECT 1 FROM active_moments m
                 WHERE m.id = e.to_id AND m.person_id = $2
           )
       )
ORDER  BY random()
LIMIT  1;
```

(Production version will pre-compute the "already answered" filter to
avoid the random + correlated subquery — but this expresses the rule.)

### Thread Detector trigger check

```sql
SELECT (cnt - p.moments_at_last_thread_run) >= 15 AS should_run,
       cnt                                        AS active_total
FROM   persons p
JOIN LATERAL (
    SELECT count(*) AS cnt FROM active_moments WHERE person_id = p.id
) c ON TRUE
WHERE  p.id = $1;
```

## 9. Operational notes

### Re-embedding on model change

When the embedding model is upgraded, re-embed in batches:

1. Pick a record set still on the old model (`WHERE embedding_model =
   <old> AND status = 'active'`).
2. For each row, push an embedding job to SQS with the **new** model
   identity in the payload.
3. The embedding worker UPDATEs the vector + model + version
   together. The completeness CHECK ensures partial state is
   impossible.

Never let two models coexist for the same `person_id` in production
similarity queries (invariant #3). Either re-embed everything or
filter by model in every query.

### HNSW maintenance

The HNSW indexes are partial (`WHERE status = 'active'`). When rows
are superseded or archived, they leave the index automatically.
Postgres `VACUUM` will reclaim the space. No special tuning needed
for v1 scale.

### Soft-delete patterns

- **Moment refinement** (Extraction Worker): old → `superseded`,
  new → `active`, ALL inbound edges repointed to new in same
  transaction. Re-embed the new row.
- **Entity merge** (user action via Node): losing → `merged`,
  `merged_into` set to surviving id, edges repointed, surviving
  re-embedded.
- **User edit on moment** (via Node): in-place UPDATE plus a
  `moment_history` row with `previous_values`. If the edited field
  is embedded, push a re-embed job.

### Indexes worth knowing


## 10. What's deliberately not here in v1

- No `entity_history`, `thread_history`, or `trait_history`. Only
  moments are user-edited heavily enough to need an audit table.
- No materialized views. We may add them post-v1 for the Profile
  Summary or hot Retrieval queries.
- No partitioning. Volume per legacy is small enough.
- No row-level security. Person scoping is enforced in app code.
- No audit of edge writes. The unique constraint + status field +
  timestamps are sufficient for v1.
